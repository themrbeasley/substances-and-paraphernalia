## 2. Resolved design decisions

These were settled during brainstorming on 2026-05-12.

### 2.1 Item 13's design intent is tier-1-to-tier-2 by default

Per Item 13 action 1's recommendation: shipped content sits in DC 12-16 (tier 1 to tier 2). Higher DCs (17+) reserved for tier-3+ setpiece substances. This matches SRD poison rules and gives the v0.9 compendium rebuild a known scaling target. Authors override per-substance.

Hint-string wording (lifted from Item 13 action 4): *"Recommended DC ranges: 12-14 for tier 1 substances, 15-17 for tier 2, 18-20 for tier 3, 21+ for tier 4 setpiece substances."*

### 2.2 Item 4's preview is a non-blocking computed span, not a form field

The withdrawal-duration preview reads `withdrawalMod` + (optionally) the editing actor's Con modifier if one is selectable, and renders the formula's output as a live-updating span beneath the Withdrawal Mod input. It is purely cosmetic — the engine continues to compute `restsRemaining` at apply time via [scripts/data/withdrawal.js](../../../scripts/data/withdrawal.js). The preview is **not** persisted; it is a UI affordance.

Display format: *"Estimated withdrawal: ~N long rests (assumes Con +0; varies per character)."* The "assumes Con +0" disclosure prevents the obvious misread that the number is fixed.

### 2.3 Item 11's split: audit-once surfaces in v0.8; rewrite-once surfaces in v0.9

`lang/en.json` and `templates/*.hbs` are stable — they don't churn during the v0.9 compendium rebuild, so auditing them now and never again is the cheap, correct cadence. `_source/**/*.json` (shipped item descriptions, AE descriptions, in-world journal) is rewritten wholesale in v0.9 / Item 12, so auditing it now is throwaway work.

The legacy-phrasing invariant ships **warn-only** in v0.8. v0.9 flips it to error-blocking once the rewrite is in.

### 2.4 Subsystem visibility uses Foundry's standard collapsed-field pattern

When "Addiction" is toggled off, the addiction-dependent fields (Save Ability, Save DC, Withdrawal Mod, Withdrawal Effect picker) collapse via `[data-fishut-toggle]` + a `.fishut-hidden` CSS class — not greyed-out, just hidden. Same pattern for Overdose, Tolerance, and the paraphernalia "Grants Save Bypass" toggle. Reasoning: grey-out reads as "broken"; hide reads as "off." The author's mental model is on/off, not enabled/disabled-but-visible.

Authors need to know the intent to tune DCs sensibly. This item is documentation only, but it informs the Item 12 compendium rebuild and any author's homebrew tuning.

2. Add a hint string under the Save DC field in [templates/details-tab/substance-fields.hbs](../../../templates/details-tab/substance-fields.hbs). Render in a `.fishut-hint` span beneath the input. Localize as `FISHUT.Details.SaveDc.Hint`.

### 3.2 Item 4 — Withdrawal duration preview + subsystem visibility

**Context (verbatim from Item 4):**

> The schema field `addiction.withdrawalMod` is a number used in the formula that computes `restsRemaining` (the count of long rests a withdrawal AE persists). The exact formula (e.g., `restsRemaining = withdrawalMod + 1d4` or `restsRemaining = withdrawalMod` flat) lives only in source code and is not documented on the wiki. Authors tuning a substance have no way to preview the resulting withdrawal duration from the Details-tab authoring surface, and the lack of clarity makes substance authoring feel like flying blind. The author has also noted that not every substance uses every option (some may have only an Altered effect with no addiction or withdrawal at all), which means the authoring surface should make it visually obvious which subsystems are configured for a given substance and which are inert.

**Scope:** Document the existing formula (no formula change). Add a live preview span. Subsystem-visibility audit across the Details tab — substance fields and paraphernalia fields.

**Out of scope:** Changing the formula. Auto-generating withdrawal AEs.

**Implementation:**

1. Document the existing formula in [scripts/data/withdrawal.js](../../../scripts/data/withdrawal.js) — add a single-line comment on the `restsRemaining` computation. Per CLAUDE.md "default to writing no comments unless the WHY is non-obvious," the comment captures the **why** of the floor clamp: *"// floor clamp prevents a high-Con character from waving off withdrawal entirely."*
2. Add "Tuning Withdrawal Duration" sub-section to [docs/wiki/Authoring.md](../../../docs/wiki/Authoring.md). Show the formula (`restsRemaining = max(withdrawalMod − ConMod, ⌈withdrawalMod/2⌉)`, min 1) and at least three worked examples (`withdrawalMod` = 2, 4, 6 against Con +0, +3, +5).
3. Add a read-only preview span in [templates/details-tab/substance-fields.hbs](../../../templates/details-tab/substance-fields.hbs) below the Withdrawal Mod input. Element: `<span class="fishut-preview" data-fishut-preview="withdrawal">…</span>`.
4. Wire the live update in [scripts/ui/details-tab.js](../../../scripts/ui/details-tab.js): on input-change of the Withdrawal Mod field, recompute the preview value via `previewWithdrawalDuration(withdrawalMod, assumedConMod = 0)` (new pure function exported from `scripts/data/withdrawal.js`). Render the formatted string into the span. Format per §2.2.
5. Subsystem-visibility audit:
   - In [templates/details-tab/substance-fields.hbs](../../../templates/details-tab/substance-fields.hbs): wrap the Addiction-dependent fields (Save Ability, Save DC, Withdrawal Mod, Withdrawal Effect picker, Withdrawal preview) in a `<div data-fishut-collapse="addiction">` block.
   - Same for Overdose (Chance %, Description, Tolerance Interaction from Item 6, Magnitude).
   - Same for Tolerance (the three knobs + caps block from Item 5).
   - In [templates/details-tab/paraphernalia-fields.hbs](../../../templates/details-tab/paraphernalia-fields.hbs): wrap the bypass-related fields in `<div data-fishut-collapse="grants-bypass">`. Add a small read-only preview *"Required for: ingested, injected"* (or whatever admins are checked) under the `appliesTo` checkbox group.
   - CSS rule: `.fishut-hidden { display: none; }`. JS in [scripts/ui/details-tab.js](../../../scripts/ui/details-tab.js) toggles the class based on the parent toggle's checked state on render and on change.
6. New localization keys in [lang/en.json](../../../lang/en.json): `FISHUT.Details.Preview.Withdrawal`, `FISHUT.Details.Preview.AppliesTo`.

**Acceptance (verbatim from Item 4):**

- The Authoring wiki page has a "Tuning Withdrawal Duration" section with the formula and worked examples.
- The Details-tab authoring surface shows a live withdrawal duration preview that updates as the author edits Withdrawal Mod.
- Subsystem-disabled fields are visually distinct from subsystem-enabled fields in the Details tab.
- Paraphernalia authoring surface has the same subsystem clarity.

**Tests:**

- **Unit** ([test/unit/withdrawal-formula.test.mjs](../../../test/unit/withdrawal-formula.test.mjs), extend existing file): test `previewWithdrawalDuration` across the worked-examples grid (`withdrawalMod` ∈ {2,4,6} × `assumedConMod` ∈ {0,3,5}). Pure function, no Foundry globals.
- **Manual verification** in a live world: subsystem toggles hide/show the dependent fields without page reload; the preview updates as the input changes; the paraphernalia `appliesTo` preview reflects the current checkbox state.

### 3.3 Item 11 (partial) — 2024 language audit, stable surfaces only

**Context (verbatim from Item 11):**

> The module currently renders text in chat cards, AE descriptions, substance descriptions, paraphernalia descriptions, and various dialogs and prompts. Much of this language likely uses pre-2024 phrasing ("once per day," "become invisible," etc.) instead of 2024-compliant phrasing ("regains all uses at dawn," "gains the Invisible condition," etc.). The author has acknowledged this and wants a full audit and rewrite pass.

**Scope (v0.8 portion):** Audit and rewrite `lang/en.json` and `templates/*.hbs`. Add a warn-only legacy-phrasing invariant to [tools/validate-content.mjs](../../../tools/validate-content.mjs). Add the "Language Conventions" sub-section to the Authoring wiki.

**Out of scope (deferred to v0.9):** Auditing `_source/**/*.json` (item descriptions, AE descriptions, in-world journal). Flipping the invariant to error-blocking.

**Implementation:**

1. Read every string in [lang/en.json](../../../lang/en.json) against the Item 11 action 2 checklist:
   - Capitalized conditions: Blinded, Charmed, Deafened, Exhaustion, Frightened, Grappled, Incapacitated, Invisible, Paralyzed, Petrified, Poisoned, Prone, Restrained, Stunned, Unconscious.
   - "make a [Ability] saving throw," not "roll a [Ability] save."
   - "regain [X] hit points," not "restore" or "recover."
   - "expend a spell slot."
   - "until the end of your next turn" / "until the start of your next turn" — used precisely.
   - Damage types lowercase. Ability scores capitalized. Rests capitalized (Long Rest, Short Rest).
   - "regains all expended charges at dawn" / "you can't use this feature again until you finish a Long Rest" replacing "once per day."
   - "gain the X condition" replacing "become X" for conditions.
2. Rewrite each non-compliant string in place. Where a chat card composes multiple strings, verify the rendered result reads cleanly.
3. Repeat the audit over every inline string in [templates/details-tab/*.hbs](../../../templates/details-tab/) and any other `templates/**/*.hbs` file. Most strings should already route through `{{localize …}}`; any that don't get moved to `lang/en.json` first, then audited.
4. Add a **warn-only** legacy-phrasing invariant to [tools/validate-content.mjs](../../../tools/validate-content.mjs). Patterns to flag:
   - `/once per day/i`
   - `/become invisible/i`, `/become poisoned/i`, `/become frightened/i`, `/become charmed/i`, `/become …/i` against any of the 15 condition tokens (case-insensitive on the condition name, but flag because the verb is "become" not "gain the")
   - `/roll a \w+ save/i`, `/make a \w+ save\b/i` (the second pattern is fine when followed by "saving throw"; the regex needs a word-boundary or negative lookahead)
   - `/restore (\d+|\w+) hit points/i` (should be "regain")
   - `/recover (\d+|\w+) hit points/i` (should be "regain")
   - Condition tokens in lowercase outside of a code block: `/\b(blinded|charmed|deafened|frightened|grappled|incapacitated|invisible|paralyzed|petrified|poisoned|prone|restrained|stunned|unconscious)\b/` — match only against text content, not against `flags.*` or status-id values.
   - Damage type names in uppercase: `/\b(Fire|Cold|Acid|Lightning|Necrotic|Radiant|Poison|Psychic|Force|Thunder)\b/` matching damage-type usage. (Be tolerant of "Poison" when it's the **subtype** keyword on a consumable — that's dnd5e's data field, not display prose. Filter the invariant to text-content paths.)
5. Run the invariant against the v0.8 codebase post-rewrite. All warns must come from `_source/**/*.json` (which v0.8 explicitly does not audit); zero warns from `lang/en.json` or `templates/*.hbs`.
6. Add "Language Conventions" sub-section to [docs/wiki/Authoring.md](../../../docs/wiki/Authoring.md). Document the checklist from action 1. Reference the SRD where applicable. State explicitly that the invariant flips to error-blocking in v0.9 and authors should bring custom content into compliance by then.

**Acceptance (v0.8 portion, derived from Item 11):**

- Every player-facing string in `lang/en.json` uses 2024 phrasing.
- Every inline string in `templates/*.hbs` uses 2024 phrasing.
- The legacy-phrasing invariant runs in `npm run validate` and emits warns (not errors).
- Post-rewrite, `npm run validate` produces zero warns sourced from `lang/en.json` or `templates/*.hbs`.
- "Language Conventions" sub-section exists on the Authoring wiki.

**Out-of-scope completion checkpoints (for v0.9):**

- `_source/**/*.json` audited.
- Invariant flipped to error-blocking.

**Tests:**

- **Unit** (new file `test/unit/validate-content-language.test.mjs`, add to `package.json` `test:unit`): test the invariant function against a battery of compliant and non-compliant strings. Verify it warns (returns warn-level findings) rather than erroring.
