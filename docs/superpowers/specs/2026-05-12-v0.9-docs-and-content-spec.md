## 2. Resolved design decisions

These were settled during brainstorming on 2026-05-12.

### 2.1 Manuals use invented example substances, not reused shipped content

Per Item 7 action 1 ("invent a clean, generic, demo-quality substance specifically for the documentation"): the manual's worked example is **invented for documentation purposes**, not a re-skinned shipped substance. Working name **"Example: Greyleaf Tea"** (substance) + **"Greyleaf Pipe"** (paraphernalia). Reasoning per Item 7: shipped compendium substances are testbed content per author feedback E; reusing them couples doc accuracy to content churn.

The example never ships in the compendium. It lives only in the wiki + README + GM Guide journal.

### 2.2 Manuals lead with tutorial, then reference

Both manuals follow the same shape:
- **Part 1 — Tutorial:** Build the example from a blank consumable / equipment item, step by step, with screenshots of every authoring decision. Reads top-to-bottom.
- **Part 2 — Reference:** One sub-section per subsystem (Altered / Addiction / Withdrawal / Tolerance / Overdose for substances; Subtype / Admin-match / Bypass / Readiness for paraphernalia). Reference sub-sections are dip-in-able by title — readers don't read them in order.
- **Part 3 — Common pitfalls:** End-of-page sidebar.

Page-length budget: aim for ~3000 words per manual; split into `-Tutorial.md` and `-Reference.md` only if a single page would exceed that.

### 2.3 README cards are short; tutorial stays on the wiki

Item 9 / Item 10 ship a **single rendered card** per item type — flavor + canonical six-section description format from v0.2 — and a one-line link to the wiki tutorial. The README does not teach authoring; it shows what authored content looks like. Per Item 9 / Item 10: "the README's job is to show, not to teach."

### 2.4 Item 11 completion flips the invariant on the same PR that finishes the audit

The legacy-phrasing invariant added in v0.8 ships as **warn-only**. v0.9 finishes the audit (every `_source/**/*.json` description rewritten or confirmed compliant) and flips the invariant to error-blocking in the same PR. Reasoning: a flipped invariant against a not-yet-audited tree would error-block every existing substance and unblock no one; an audited tree with a still-warn invariant gives no guarantee that future content stays clean. Both must land together.

### 2.5 Item 12 has a roster shell in v0.9, but the per-substance plan is deferred to v0.9 kickoff

This v0.9 spec defines:
- The matrix shape (3×3 setting × category, **at least 2 substances + at least 1 paraphernalia per cell**).
- The invented vs reused boundary (manuals invent; compendium can keep recognizable names where the rebuild upgrades the content rather than redesigning it).
- The authoring conventions every rebuilt item must satisfy.

This v0.9 spec **does not** define:
- Which specific substances ship in each cell (kept or retired).
- Per-substance Altered / Addiction / Withdrawal / Tolerance / Overdose values.

Rationale: v0.7 + v0.8 playtest will surface tuning lessons (tolerance caps in practice, overdose × tolerance feel, voluntary-abstain consequences). Locking per-substance detail now is premature. **Write the Item 12 sub-spec at v0.9 kickoff** (`docs/superpowers/specs/<v0.9-kickoff-date>-item-12-compendium-rebuild-subspec.md`), once v0.8 is in users' hands.

The roster shell is enough to schedule the release weight; the sub-spec is enough to schedule the PR.

## 3. Per-item requirements

### 3.1 Item 7 — Substance authoring manual

**Context (verbatim from `tasks/SP-MODULE-TODO-2026-05-12.md` Item 7):**

> The author has identified that the current documentation does not constitute a "GM-ready instruction manual" for authoring substances. A GM picking up the module today cannot easily learn how to create a new substance from scratch, with all subsystems (Altered effect, Addiction, Withdrawal, Tolerance, Overdose) properly configured. The fix is a deliberate end-to-end authoring manual that walks a new GM through creating one specific worked example, then provides reference material for each subsystem.

**Scope:** New wiki page (or 2-page split per §2.2 budget). Tutorial-then-reference. Single invented worked example (Greyleaf Tea).

**Out of scope:** Updating the in-world journal beyond the single pointer page (the journal stays a short pointer to the wiki). Mechanical changes — manual reflects what v0.7 + v0.8 shipped; it does not redesign.

**Implementation:**

1. Build the Greyleaf Tea example end-to-end in a documentation session, capturing every step. Verbatim from Item 7 action 2:
   - Create a new consumable item.
   - Toggle "Illicit Substance" in the Details tab.
   - Fill in Setting, Category, Administration.
   - Author the Altered AE (the benefit effect): include 2024-compliant prose template for the AE description.
   - Author the Addiction AE template: include 2024-compliant prose template.
   - Author the Withdrawal AE template: include 2024-compliant prose template, with guidance on what NOT to put in (the existing content invariant about not duplicating Poisoned).
   - Fill in Save Ability, Save DC, Withdrawal Mod.
   - Configure Tolerance (or note that this substance does not use tolerance and explain when to skip).
   - Configure Overdose (or note that this substance does not use overdose and explain when to skip).
   - Configure Required Paraphernalia (or note none required).
   - Write the substance description text using the canonical six-section format from v0.2 (flavor, desired effects, save against addiction, addicted-from, withdrawal mod and formula, requires footer).
2. Take screenshots of every Details-tab step. The wiki supports images.
3. Reference sub-sections (verbatim from Item 7 action 4):
   - **Altered AE reference:** AE Changes table conventions, TMFX `macro.tokenMagic` Change row pattern (cross-reference CLAUDE.md §"TMFX integration is DAE-driven, not a custom hook"), recommended duration formats.
   - **Addiction AE reference:** poisoned-coupling interaction, when to attach Poisoned via the AE vs. let the engine do it.
   - **Withdrawal AE reference:** vignette color authoring (AE Change row, mode 5 OVERRIDE, key `flags.substances-and-paraphernalia.vignetteColor`), don't-duplicate-Poisoned guidance.
   - **Tolerance reference:** the three knobs, the bounds from v0.7 / Item 5 (with cross-link to the wiki section that documents them), when to leave tolerance off entirely.
   - **Overdose reference:** `chancePercent` semantics, description-only chat card authoring, tolerance interaction from v0.7 / Item 6.
4. **Common pitfalls sidebar** (verbatim from Item 7 action 5):
   - Forgetting to set the save ability for mind-altering substances (cross-link Item 1).
   - Leaving a withdrawal AE template empty.
   - Setting Tolerance knobs without thinking about cumulative bite.
   - Accidentally making consumption a pure-loss.
5. Cross-reference Item 13's DC-scaling sub-section (v0.8) inside the Save DC tutorial step.
6. Page lives at [docs/wiki/Authoring-Substances.md](../../../docs/wiki/Authoring-Substances.md). If word count crosses ~3000, split into `-Tutorial.md` and `-Reference.md`.

**Acceptance (verbatim from Item 7):**

- A wiki page titled something like "Authoring Substances: Step-by-Step" exists.
- The page walks through a complete example substance build from blank item to finished.
- Every authoring subsystem has a reference sub-section.
- A "common pitfalls" sidebar exists.
- Screenshots of the Details-tab fields are present.

**Tests:** No code-level test for documentation. Verify by reading: a new contributor with zero prior context can follow the tutorial and produce a working substance in a live world.

### 3.2 Item 8 — Paraphernalia authoring manual

**Context (verbatim from Item 8):**

> Paraphernalia gets less documentation attention than substances because the substance is the primary mechanical surface. But paraphernalia is half the system: gating, save bypass, optional consumable-uses-driven readiness, the AE-flag bypass pattern. A GM authoring custom paraphernalia today has the same blind-flying problem as a GM authoring substances. This item is the paraphernalia equivalent of Item 7.

**Scope:** New wiki page, same shape as Item 7. Single invented worked example (Greyleaf Pipe, paired with the Greyleaf Tea from Item 7).

**Implementation:**

1. Build the Greyleaf Pipe example end-to-end (verbatim from Item 8 action 1):
   - Create a new equipment item (or consumable, if the example needs one-shot uses).
   - Toggle "Paraphernalia" in the Details tab.
   - Fill in Setting, Paraphernalia ID (subtype), `appliesTo` admin-type checkboxes.
   - If the example paraphernalia grants a save bypass: author the bypass-granting AE on the paraphernalia item, with the `modifier.kind: "bypass"` flag and the appropriate `type` field (`auto-pass`, `reroll-on-fail`, `advantage`, `+N`).
   - Also write `aeRole: "bypass"` on the AE flag block (per v0.7 / Item 2).
   - Write the paraphernalia description text in 2024-compliant prose, with charges-and-recharge phrasing where applicable.
2. Reference sub-sections (verbatim from Item 8 action 2):
   - **Subtype model:** the kebab-case open enum, the Paraphernalia Subtype Manager settings sub-menu for custom subtypes (cross-link [docs/wiki/Paraphernalia-Subtype-Manager.md](../../../docs/wiki/Paraphernalia-Subtype-Manager.md)).
   - **Admin-type matching:** how `appliesTo` interacts with the dnd5e Poison subtype on substances. Cross-link the CLAUDE.md "Admin-type gate" section's behavior.
   - **Save bypass authoring:** the four tiers (`auto-pass > reroll-on-fail > advantage > +N`), when to pick each, `usesPerDay` semantics, the resolution order in the modifier pipeline (cross-link [docs/wiki/Save-Bypass-Tiers.md](../../../docs/wiki/Save-Bypass-Tiers.md)).
   - **Readiness gating:** paraphernalia that requires being equipped, paraphernalia that requires uses remaining, paraphernalia that requires both.
3. **Common pitfalls sidebar** (verbatim from Item 8 action 3):
   - Forgetting to set `appliesTo`.
   - Authoring a bypass AE without `transfer: true` so it doesn't move to the actor.
   - Picking a save bypass tier higher than the substance design warrants.
4. Page lives at [docs/wiki/Authoring-Paraphernalia.md](../../../docs/wiki/Authoring-Paraphernalia.md). Split as in Item 7 if needed.

**Acceptance (verbatim from Item 8):**

- A wiki page titled something like "Authoring Paraphernalia: Step-by-Step" exists.
- The page walks through a complete example paraphernalia build.
- Every authoring subsystem (subtype, admin-type, save bypass, readiness) has a reference sub-section.
- Common pitfalls sidebar exists.
- Screenshots present.

**Tests:** Same as Item 7 — read-verify.

### 3.5 Item 12 — Compendium rebuild (roster shell)

**Context (verbatim from Item 12):**

> The author has acknowledged that the current shipped compendium content is testbed material and will not ship in v1.0 form. The compendium rebuild is the right vehicle for applying every documentation-driven authoring lesson learned in Items 7 through 11. The rebuild also benefits from the bounded Tolerance system (Item 5), the overdose-tolerance interaction (Item 6), and the per-category save ability defaults (Item 1).

**Scope:** Rewrite every shipped substance and paraphernalia item to:
- Use 2024 phrasing (Item 11 completion).
- Apply per-category save abilities (Item 1's shipped-content portion — mind-altering substances switch to Wisdom; stimulants and performance-enhancers stay Constitution; at least one explicit per-substance override exists).
- Apply the v0.8 DC-scaling intent (Item 13 — shipped content sits in DC 12-16 by default).
- Carry `aeRole` flags on every AE (Item 2 — already invariant-checked from v0.7).
- Carry `tolerance.caps` overrides only where the substance is intentionally a severe-progression design (Item 5).
- Carry `overdose.toleranceInteraction` per the author's design choice for each overdose-capable substance (Item 6).
- Pass every content invariant including the post-flip error-blocking legacy-phrasing invariant (Item 11 completion).

**Out of scope:** Expanding the matrix beyond 3×3. Adding settings or admin categories.

**Roster shell:**

- 3×3 matrix = 9 cells (`{fantasy, modern, sci-fi} × {stimulant, performance-enhancer, mind-altering}`).
- **At least 2 substances per cell** (one "core" substance exercising the standard Altered + Addiction + Withdrawal loop; one "variant" exercising a different combination — e.g., no addiction, just tolerance-progressive Altered; or required-paraphernalia gating example).
- **At least 1 paraphernalia per cell** that pairs thematically with the substances in that cell.
- At least one substance across the compendium overrides the per-category save default to demonstrate the override path (per Item 1 action 1).

**Per-substance plan deferred to v0.9 kickoff sub-spec** (per §2.5).

**Files affected:**

- All of [_source/fishut-illicit-substance/](../../../_source/fishut-illicit-substance/) — rewrite or retire each substance JSON.

- [README.md](../../../README.md) and the GM Guide journal for re-import guidance (per Item 12 action 6).

**Acceptance (verbatim from Item 12):**

- Every cell of the 3×3 matrix has at least 2 substances and 1 paraphernalia item.
