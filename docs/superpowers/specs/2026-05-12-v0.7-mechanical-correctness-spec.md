## 2. Resolved design decisions

These were settled during brainstorming on 2026-05-12. Implementers should not re-litigate.

### 2.1 Item 3 implementation seam = `activity.use()`, not synthetic hook

The abstain failure path must run the substance through its **real** post-use chain (save ‚Üí addiction AE ‚Üí tolerance stack ‚Üí overdose roll). The cleanest seam is the same one the "Use anyway" override already uses: register the activity ID in `activity-gating.js#bypassOnce`, then call `activity.use()` on the substance's primary activity. Foundry's activity machinery emits `preUseActivity` (gate consumes the bypass and lets the activity through) ‚Üí `postUseActivity` (addiction.js + overdose.js fire normally).

Rejected alternatives:
- **Synthesizing a `dnd5e.postUseActivity` Hooks.callAll** ‚Äî bypasses Foundry's own use-side-effects (item-uses decrement, chat-card emission, activity-cost handling). Tests would pass; the in-world behavior would silently drop a use.
- **Calling `rollSaveAndApply` directly** ‚Äî addiction.js exports it as a test seam, but it skips the overdose listener entirely and skips the activity's own chat card. Inconsistent with how a real consumption looks.

### 2.2 Item 2 fallback retention is permanent, not a deprecation

The substring matcher (`addict`, `withdraw`, `overdose`, `tolerance`, `altered`) stays as a warn-logged fallback after v0.7. Out-of-scope explicitly: removing the fallback. Reason: GMs hand-author AEs in their own worlds without going through the module's templates; we cannot retroactively flag them. Warn-log makes the fallback observable; never silent.

### 2.3 Item 5 caps are engine-applied soft defaults, never validator-blocking

The schema accepts any cap value (or omits the `caps` block entirely, in which case the engine defaults apply). `tools/validate-content.mjs` emits a **warn** when an author-supplied cap exceeds the engine default; never an error. Authors who want runaway tolerance opt in by writing the override explicitly.

### 2.4 Item 6 reads tolerance stacks at roll time, not at AE-apply time

The d100 overdose roll modulates against the actor's **current** tolerance stack count for the substance, read at the moment `rollOverdoseAndApply` fires. Reading at apply time would cache the wrong value (the addiction AE is what increments tolerance, and it fires in the same `postUseActivity` cycle ‚Äî order of two independent listeners is not guaranteed).

Resolution: overdose's listener reads stacks from `actor.appliedEffects` (via the `aeRole: "tolerance"` flag once Item 2 lands, substring fallback before). If the listener fires before addiction has applied the new tolerance stack, the d100 sees the **prior** stack count. This is fine ‚Äî the new stack only takes effect from the next consumption forward, which matches the "tolerance grows over time" mental model. Codify this in a code comment.

### 2.5 Item 1 split across v0.7 and v0.9

The schema continues to allow any save ability per substance. v0.7 ships the hint string + Authoring-wiki sub-section + GM Guide pointer note (cheap to author, zero content churn). The shipped-content sweep (every mind-altering substance switches to Wisdom) waits for v0.9 / Item 12 ‚Äî the compendium is being rewritten then anyway, and pre-touching content now is throwaway work.

## 3. Per-item requirements

### 3.1 Item 2 ‚Äî `aeRole` flag

**Context (verbatim from `tasks/SP-MODULE-TODO-2026-05-12.md` Item 2):**

> The module currently identifies what an Active Effect "is" by substring-matching the AE's name (`addict`, `withdraw`, `overdose`, `tolerance`, `Altered by *`). This works in a single-locale, single-author world and breaks the moment a French GM renames an AE to its local-language equivalent, an author makes a typo, or a homebrewer names a bypass AE something like "Anti-addiction filter" and gets it false-positive-matched as an addiction AE. The module already uses AE flags for some semantic tagging (`sourceSubstanceId` from v0.2, `flags[...].modifier` from v0.3). Extending that pattern with an explicit role tag, while keeping the substring matcher as a warn-logged fallback, gives locale-independence and typo-tolerance without breaking hand-authored AEs.

**Scope:** A new flag `flags["substances-and-paraphernalia"].aeRole` with enum values `"addiction" | "withdrawal" | "altered" | "tolerance" | "overdose" | "bypass"`. Read it from macros, sweeps, the modifier pipeline, and every site currently doing substring matches. Write it on every AE the module creates or templates. Fall back to substring match when the flag is absent and warn-log when that fallback fires.

**Out of scope:** Removing the substring matcher entirely. The fallback stays permanently per ¬ß2.2.

**Implementation:**

1. Append the `aeRole` enum to [scripts/data/schema.json](../../../scripts/data/schema.json) under a new top-level `aeRoles` block (label keys: `FISHUT.AERole.Addiction`, `‚Ä¶Withdrawal`, `‚Ä¶Altered`, `‚Ä¶Tolerance`, `‚Ä¶Overdose`, `‚Ä¶Bypass`).
2. Every AE creation site writes `aeRole` on the flag block:
   - [scripts/hooks/addiction.js](../../../scripts/hooks/addiction.js) `applyAddictionEffect` ‚Üí `aeRole: "addiction"`.
   - [scripts/hooks/addiction.js](../../../scripts/hooks/addiction.js) `applyOrIncrementToleranceStack` ‚Üí `aeRole: "tolerance"` (on first-stack create; the increment path mutates the existing AE, no re-write needed).
   - [scripts/hooks/addiction.js](../../../scripts/hooks/addiction.js) `applyWithdrawalEffect` ‚Üí `aeRole: "withdrawal"`. The default template builder also writes it.
   - [scripts/hooks/overdose.js](../../../scripts/hooks/overdose.js) `applyOverdoseEffect` (if/when an overdose marker AE is applied) ‚Üí `aeRole: "overdose"`.
   - `_source/fishut-illicit-substance/*.json` benefit-AE templates (existing `Altered by ‚Ä¶` AEs) ‚Üí `aeRole: "altered"`. Rebuild via `npm run pack`.
   - `_source/fishut-illicit-paraphernalia/*.json` bypass-AE templates ‚Üí `aeRole: "bypass"`. `modifier.kind: "bypass"` continues to live alongside; the two are redundant but explicit.
3. AE lookup sites switch to flag-first, substring-fallback. A single helper `findEffectsByRole(actor, role)` lives in [scripts/data/flag-schema.js](../../../scripts/data/flag-schema.js) (since that's the canonical flag I/O surface). Sites that switch:
   - [scripts/data/modifier-pipeline.js](../../../scripts/data/modifier-pipeline.js) bypass walker (currently keys off `modifier.kind`, which already works for module-created AEs; adding `aeRole` is a belt-and-suspenders correctness check and unblocks hand-authored bypass AEs that have `aeRole` but no `modifier.kind`).
   - `_source/fishut-illicit-macros/*.json` Remove-X macros (Remove Addiction, Remove Withdrawal, Remove Altered, Remove Tolerance, Remove Overdose). Each currently regexes the AE name; switch to `findEffectsByRole` with regex fallback. Rebuild via `npm run pack`.
   - [scripts/integrations/dae.js](../../../scripts/integrations/dae.js) ‚Äî unaffected; it scans `effect.changes`, not names.
4. Warn-log the fallback: when `findEffectsByRole` finds nothing via flag and resorts to substring matching, emit `logger.warn("aeRole flag missing on AE matched by substring fallback", { actorId, effectId, role })`. GMs see it in the console; we have an observable signal that hand-authored AEs are in play.
5. Add a content invariant in [tools/validate-content.mjs](../../../tools/validate-content.mjs): every AE in `_source/**/*.json` whose name contains one of the role substrings (case-insensitive) must carry the matching `aeRole` flag. Error-level, not warn.
6. Wiki: add a new "AE Conventions" sub-section to [docs/wiki/Authoring.md](../../../docs/wiki/Authoring.md) explaining the flag, the role enum, and the substring-fallback warn behavior.

**Acceptance (verbatim from Item 2):**

- Every shipped compendium AE carries an `aeRole` flag and passes the new content invariant.
- All five Remove-X macros find their target AEs via `aeRole` and only fall back to substring matching when the flag is missing.
- The modifier pipeline reads bypass AEs via flag, not via name.
- A Quench test renames an addiction AE to a non-matching string (e.g., "Toxisch durch Kohlenschattenpulver") and confirms the Remove Addiction macro still finds it via the flag.
- A console warning is emitted (and visible to GMs) when the substring fallback fires.

**Tests:**

- **Unit** (new file `test/unit/ae-role-lookup.test.mjs`, add to `package.json` `test:unit`): pure-function tests on `findEffectsByRole` ‚Äî flag-only match, substring-only match, both, neither.
- **Quench** (new test in [test/quench/test-suite.mjs](../../../test/quench/test-suite.mjs), suite name `S&P ¬∑ aeRole ‚Äî renamed AE still removable`): apply an addiction AE, rename it via `effect.update({ name: "Toxisch ‚Ä¶" })`, invoke the Remove Addiction macro, assert the AE is removed.
- **Quench** (suite name `S&P ¬∑ aeRole ‚Äî hand-authored AE without flag warn-logs`): create an AE named `Withdrawal from X` directly (no `aeRole` flag), invoke Remove Withdrawal macro, assert removal + warn-log captured via a `logger.warn` spy.

### 3.3 Item 5 ‚Äî Tolerance soft caps

**Context (verbatim from Item 5):**

> The Tolerance system has three knobs (attenuateAltered, addictionDcBump, withdrawalAmplify) that all make the next consumption worse: weaker buff, harder save, nastier withdrawal. Without bounds, an author can write a substance whose Tolerance progression eventually produces a state where the buff is fully attenuated (effectively zero benefit) and withdrawal is amplified beyond the most punishing baseline withdrawal in the compendium. The author's design intent is that Tolerance produces diminishing returns and a sense of cost, not a state where consumption is mathematically a pure loss. Some of this is documentation (authors are told not to write substances that progress to extreme states), some can be codified as soft caps in the engine.

**Scope:** Engine clamps; per-substance overrides allowed; warn-only validator. No hard validator blocking. No mechanical change to the three knobs themselves.

**Implementation:**

1. Extend [scripts/data/schema.json](../../../scripts/data/schema.json)'s tolerance block with an optional `caps` sub-object:
   ```json
   "caps": {
     "maxStacks": 5,
     "modifierFactorFloor": 0.25,
     "addictionDcBumpCap": 5,
     "withdrawalDurationFactorCap": 2.0
   }
   ```
   These are the **engine defaults**; per-substance authoring may override any subset. Missing entries fall back to defaults.
2. Update [scripts/data/tolerance.js](../../../scripts/data/tolerance.js) to apply caps at compose time:
   - `maxStacks` clamps the stack count when `applyOrIncrementToleranceStack` would push past it (the increment becomes a no-op, not a roll-over).
   - `modifierFactorFloor` clamps `attenuateAltered.modifierFactor` lookups so the buff cannot drop below the floor.
   - `addictionDcBumpCap` clamps the cumulative `addictionDcBump` total (per-stack accumulation √ó current stacks).
   - `withdrawalDurationFactorCap` clamps the cumulative `withdrawalAmplify.durationFactor`.
3. Add a warn-only invariant in [tools/validate-content.mjs](../../../tools/validate-content.mjs): if a substance's `tolerance.caps` block contains any value that exceeds the engine default (e.g., `maxStacks: 10`), emit a `warn` with the substance id, the field, the override value, and the engine default. Never error.
4. Add a "Tolerance: Bounds and Authoring Guidance" sub-section to [docs/wiki/Mechanics.md](../../../docs/wiki/Mechanics.md). Document the defaults, the override path, and the design rationale ("diminishing returns, never a pure loss").

**Acceptance (verbatim from Item 5):**

- The engine clamps tolerance stack effects at the documented caps.
- Authors can override caps per substance.
- The Mechanics wiki page documents the caps and the design rationale.
- A content invariant warn-logs when an author overrides a cap to exceed engine defaults.

**Tests:**

- **Unit** ([test/unit/tolerance.test.mjs](../../../test/unit/tolerance.test.mjs), extend or new file added to `package.json` `test:unit`): for each cap, write a pure-function test that pushes the input past the default and asserts clamping. Also a test that verifies per-substance overrides actually loosen the cap.

### 3.4 Item 6 ‚Äî Overdose √ó tolerance interaction

**Context (verbatim from Item 6):**

> The author's response to the original review point about overdose-vs-tolerance compounding raised a real design question: in some substances, building tolerance should reduce overdose risk (the body has adapted), while in others, building tolerance should increase overdose risk (users chase the diminishing buff with higher doses). The current schema treats overdose as a fixed per-use d100 chance with no tolerance interaction. The right shape is an authorable per-substance field that lets the GM author choose the interaction model for that specific substance.

**Scope:** Per-substance `overdose.toleranceInteraction` field with `none | mitigate | compound` enum + a magnitude in percentage-points-per-stack. Engine reads stacks at roll time (per ¬ß2.4) and modulates the d100 chance. Authoring surface exposes the dropdown + magnitude. No change to overdose severity or the description-only chat card.

**Implementation:**

1. Extend [scripts/data/schema.json](../../../scripts/data/schema.json) `overdose` block with:
   ```json
   "toleranceInteraction": "none" | "mitigate" | "compound",
   "toleranceInteractionMagnitude": 0   // pct-pts per stack; default 0
   ```
   Default is `"none"` for backwards compatibility with all existing shipped overdose blocks.
2. Update [scripts/hooks/overdose.js](../../../scripts/hooks/overdose.js) `rollOverdoseAndApply`:
   - Read the actor's tolerance-stack count for the substance via `findEffectsByRole(actor, "tolerance")` (post-Item 2) filtered by `sourceSubstanceId === item.id`.
   - Compute `adjustedChance` = `block.chancePercent + (stacks √ó magnitude √ó sign)` where `sign = -1` for `mitigate`, `+1` for `compound`, `0` for `none`.
   - Clamp `adjustedChance` to `[0, 100]`.
   - Roll d100 against `adjustedChance`. Inline comment cross-references ¬ß2.4's read-at-roll-time rationale.
3. Add an Overdose sub-section in [templates/details-tab/substance-fields.hbs](../../../templates/details-tab/substance-fields.hbs) exposing the dropdown + magnitude input. Hint string (lifted from Item 6 actions): *"Mitigate models physical adaptation (body tolerates the substance better, overdose risk falls). Compound models dose-escalation behavior (tolerant users seek higher doses, overdose risk rises). None is the default for substances where tolerance and overdose are mechanically unrelated."* Localize as `FISHUT.Details.Overdose.ToleranceInteraction.Hint`.
4. Document the interaction in [docs/wiki/Mechanics.md](../../../docs/wiki/Mechanics.md). Cross-reference ¬ß3.3 ‚Äî caps from Item 5 bound the stack count, which bounds the modulation envelope.

**Acceptance (verbatim from Item 6):**

- The authoring surface exposes the interaction dropdown.
- The engine applies the interaction correctly at the consumption hook.
- Unit tests pass for all three modes.
- The Mechanics wiki page documents the design choices.

**Tests:**

- **Unit** (new file `test/unit/overdose.test.mjs`, add to `package.json` `test:unit`): export `computeAdjustedOverdoseChance(baseChance, stacks, mode, magnitude)` as a pure function from `scripts/data/overdose-interaction.js`. Test all three modes, the [0,100] clamp, magnitude-zero degenerate case, stacks-zero degenerate case.
- **Quench** (extend [test/quench/test-suite.mjs](../../../test/quench/test-suite.mjs)): pre-stage an actor with 3 tolerance stacks, substance with `mitigate` and magnitude 10; trigger `postUseActivity`; stub the d100 to a value that distinguishes adjusted vs unadjusted; assert the adjusted path was taken.

### 3.5 Item 3 ‚Äî Voluntary Abstain failure path

**Context (verbatim from Item 3):**

> The SPEC describes Voluntary Abstain as "while a withdrawal AE is active, the long-rest dialog offers an Abstain this rest button. Wis save, DC = 8 + withdrawalMod. Success advances the withdrawal counter by 2. Failure is normal 1-rest progress, no penalty." The author's intended design was different: the Wisdom save represents using willpower to resist the urge to consume the substance during a craving moment (typically the long rest, when the body is asking for it). The current implementation has no teeth on a failed save because there is no negative consequence. The intended consequence of a failed abstain check is that the character gives in, locates and consumes the substance, and re-triggers the post-use addiction loop (Constitution save against the substance's DC, full consequence chain). This is both a narrative reframing and a real mechanical change to the failure path.

**Scope:** Narrative reframing (button label, chat card text ‚Üí willpower/craving) + mechanical change (failed Wis save triggers consumption via the substance's primary activity, run through `bypassOnce` ‚Üí `activity.use()`). Soft-fail when the substance is not in inventory or has zero uses remaining.

**Out of scope:** Removing the abstain mechanic. Re-tuning the DC formula (held until post-implementation playtest ‚Äî see Item 3 action 5).

**Implementation:**

1. Update [scripts/hooks/long-rest-abstain.js](../../../scripts/hooks/long-rest-abstain.js) `processAbstainSave`:
   - Today's failure branch posts `FISHUT.LongRestAbstain.Fail` and returns. Replace that with a `processAbstainFailure(actor, row)` call.
2. New `processAbstainFailure(actor, row)`:
   - Resolve the substance item: `const item = actor.items.get(row.substanceId)`. If missing, post `FISHUT.LongRestAbstain.FailNoSubstance` (new key ‚Äî "{actor} looked for {item} but had none to find") and return.
   - Resolve the substance's primary activity: `const activity = item.system.activities.contents.find(a => a.type === "utility" || a.type === "save") ?? item.system.activities.contents[0]`. If no activity, post the same `FailNoSubstance` chat card. (Defensive: a substance with no activity cannot be consumed.)
   - Check uses: if `item.system.uses?.value === 0`, post `FailNoSubstance` and return. (Substance present but exhausted ‚Äî same soft-fail as missing.)
   - Register the activity ID in [scripts/hooks/activity-gating.js](../../../scripts/hooks/activity-gating.js) `bypassOnce` via a new exported helper `registerForcedUseBypass(activityId)`. Reason: the abstain failure path is the **user** giving in; we never want the gate to block the consumption mid-flow.
   - Post `FISHUT.LongRestAbstain.FailGiveIn` ("{actor} gave in to the craving for {item}") chat card.
   - Call `await activity.use({ event: null }, { fastForward: true, chatMessage: true })`. Foundry's activity machinery emits `preUseActivity` (gate consumes bypass, lets activity through) ‚Üí `postUseActivity` (addiction.js + overdose.js run their normal flow).
3. Update the long-rest dialog button label in `promptCombinedAbstain` from "Abstain this rest" to a per-substance "Resist the urge to use {item}" label. Wording adapts per row.
4. Update chat strings in [lang/en.json](../../../lang/en.json):
   - Keep `FISHUT.LongRestAbstain.Pass` but reword to willpower language: *"{actor} resisted the urge to use {item} (DC {dc} Wisdom save). Withdrawal eases by 2 rests."*
   - Replace `FISHUT.LongRestAbstain.Fail` semantics ‚Äî rename to `FISHUT.LongRestAbstain.FailGiveIn`: *"{actor} failed to resist the craving for {item} (DC {dc} Wisdom save). The substance is consumed."*
   - Add `FISHUT.LongRestAbstain.FailNoSubstance`: *"{actor} reached for {item} but found none. The withdrawal continues."*
5. Update [docs/wiki/Voluntary-Abstain.md](../../../docs/wiki/Voluntary-Abstain.md) to describe the new failure path. Update [docs/wiki/Mechanics.md](../../../docs/wiki/Mechanics.md) cross-reference if one exists.
6. **DC re-examination is deferred.** Item 3 action 5 (consider escalating DC with consumption count) is **explicitly held** until post-v0.7 playtest. Document this hold in the CHANGELOG entry.

**Acceptance (verbatim from Item 3):**

- The long-rest dialog button label is willpower-themed, not abstain-themed.
- Failed Wisdom save triggers automatic substance consumption when the substance is in inventory.
- Failed Wisdom save with no substance in inventory soft-fails to normal 1-rest progress.
- Chat card text uses craving-and-willpower language throughout.
- Mechanics wiki page describes the corrected mechanic.
- Quench test passes.

**Tests:**

- **Quench** (new test in [test/quench/test-suite.mjs](../../../test/quench/test-suite.mjs), suite name `S&P ¬∑ Voluntary Abstain ¬∑ fail triggers consumption`):
  - Pre-stage an actor with: a withdrawal AE active; the substance item in inventory with 1 use; Wisdom set low enough that DC 8 + withdrawalMod is unreachable (or stub the save roll).
  - Trigger `preRestCompleted` with `longRest: true`.
  - Confirm via the dialog (or by direct call into `processAbstainSave` if the dialog can be bypassed in test) that the failed save fires `processAbstainFailure`.
  - Assert: `system.uses.value` decremented by 1 (consumption fired), addiction-save chat card emitted, addiction or tolerance AE updated as appropriate.
- **Quench** (new test, suite name `S&P ¬∑ Voluntary Abstain ¬∑ fail with no inventory soft-fails`):
  - Same setup, but remove the substance from inventory before the rest.
  - Trigger the same flow.
  - Assert: `FailNoSubstance` chat card emitted; no consumption fired; withdrawal continues on the standard -1 tick.
