# v1.0 — Stabilize and submit

**Status:** Approved 2026-05-12
**Thesis:** Stabilize the schema, codify release-hygiene gates, publish the content warning, and submit the module to the Foundry package registry. No mechanical changes, no schema changes, no compendium changes — v0.7 through v0.9 did that work.
**Roadmap pointer:** [2026-05-12-v0.7-to-v1.0-roadmap.md](./2026-05-12-v0.7-to-v1.0-roadmap.md)

## 1. Items in scope

In PR-landing order (rationale in §5):

1. **Item 16** — CI gate that fails a release tag without a matching `## [X.Y.Z]` CHANGELOG section.
2. **Item 14** — CSS withdrawal-vignette scoping verification + cross-module live-world test.
3. **Item 15** — `docs/SCHEMA-POLICY.md` codifying additive-only schema changes from v1.0 forward.
4. **Item 17** — Content-warning + safety-tools language in the README and GM Guide, plus a manual Foundry package-registry submission at release time.

## 2. Resolved design decisions

These were settled during brainstorming on 2026-05-12.

### 2.1 Item 14 is verification-first, refactor only on failure

The current CSS already scopes the vignette: [styles/withdrawal-vignette.css](../../../styles/withdrawal-vignette.css) writes rules against `#interface > .fishut-vignette` (a scoped descendant class), not against `#interface` itself. The mounting code in [scripts/ui/withdrawal-vignette.js](../../../scripts/ui/withdrawal-vignette.js) injects a child element under `#interface` and toggles it via `data-active`. The spec's primary task is **verification under a foreign UI module**, not a speculative refactor.

If the cross-module test surfaces a real collision (e.g., a theme that overrides `#interface > *` selectors), the fix is a more specific selector (`#interface > .fishut-vignette[data-active]` or an explicit element-id `#fishut-vignette` mounted directly under `<body>`). Do not refactor preemptively.

### 2.2 Item 15 is documentation only; no code change

The schema-freeze policy is a written commitment, not an enforcement mechanism. [scripts/migrations.js](../../../scripts/migrations.js) `MIGRATORS` stays empty per CLAUDE.md and roadmap §7. The policy doc and a `CONTRIBUTING.md` link are the entire deliverable.

The policy commits to **additive-only** changes from v1.0. Renames and removals require a major-version bump (v2.0). Existing sheet-level read-with-default code paths in `scripts/data/flag-schema.js` continue to be the migration mechanism — readers tolerate missing fields by substituting defaults. No document-level migrator is introduced.

### 2.3 Item 16's CI gate runs on tag push, fails the workflow

The release workflow already runs on tag push (`.github/workflows/release.yml` line 4-6). Add a pre-`Build module.zip` step that extracts the version from `GITHUB_REF_NAME` (stripping the `v` prefix), greps `CHANGELOG.md` for `## [<version>]`, and exits non-zero with a clear error if not found. This gate fires **before** any release artifact is produced — a failed gate produces no release, no zip, no tag asset.

CI on PRs (the `ci.yml` workflow) is **out of scope** for the gate. PRs may legitimately land without CHANGELOG entries (e.g., docs-only); the entry consolidates at release time. The gate is tag-time only.

### 2.4 Item 17 splits content-warning text (v1.0 PR) from Foundry submission (release-time manual step)

The README + GM Guide content-warning text is a normal v1.0 PR. The actual package-registry submission happens at v1.0 release time and is a **manual step** by the maintainer:

1. Re-read https://foundryvtt.com/packages/ submission guidelines at submission time (Foundry's guidelines may evolve between now and v1.0).
2. Apply whatever `module.json` content-flag or age-flag is required at that time.
3. Submit through Foundry's package-submission flow.

Pre-committing to a specific `module.json` change now would be guessing. The spec captures the content-warning text and the safety-tools recommendation; the submission mechanics ride on whatever Foundry's process looks like in 2026 onward.

### 2.5 v1.0 release stamp consumes everything from v0.7-v0.9

The stamp itself happens after Items 14-17 ship. The CHANGELOG `## [1.0.0]` entry references the schema-freeze adoption (Item 15) and the compendium-rebuild completion (already in `## [0.9.0]`), but v1.0 does not re-document v0.9's work. The release-notes scope is "release hygiene + submission," not "summary of the road since v0.6."

## 3. Per-item requirements

### 3.1 Item 16 — CHANGELOG CI gate

**Context (verbatim from `tasks/SP-MODULE-TODO-2026-05-12.md` Item 16):**

> The v0.5.2 backfilled CHANGELOG entries for v0.4.0 and v0.5.0, which indicates the release process did not require a CHANGELOG entry at tag time. A simple CI check (fail the workflow if a tag is created without a matching `## [X.Y.Z]` section in the CHANGELOG) prevents recurrence.

**Scope:** A pre-pack step in [.github/workflows/release.yml](../../../.github/workflows/release.yml) that runs after `Install` and before `Pack compendia`, extracts the version from `github.ref_name`, and greps `CHANGELOG.md` for the matching `## [<version>]` section. Fails the workflow with a clear error if not found.

**Out of scope:** Auto-generating CHANGELOG entries. Gating the CI workflow (`ci.yml`) on PR — PRs may legitimately land without CHANGELOG entries.

**Implementation:**

1. Add a new step to [.github/workflows/release.yml](../../../.github/workflows/release.yml), positioned after `Unit tests` and before `Pack compendia`:
   ```yaml
   - name: Verify CHANGELOG entry exists for tag
     run: |
       VERSION="${GITHUB_REF_NAME#v}"
       if ! grep -qE "^## \[${VERSION}\]" CHANGELOG.md; then
         echo "::error::CHANGELOG entry missing for tag ${GITHUB_REF_NAME}. Add a '## [${VERSION}]' section to CHANGELOG.md before tagging."
         exit 1
       fi
       echo "CHANGELOG entry found for ${GITHUB_REF_NAME}."
   ```
2. Document the gate in [CONTRIBUTING.md](../../../CONTRIBUTING.md) under a new "Release process" sub-section (create the sub-section if absent). Reference [docs/SCHEMA-POLICY.md](../../../docs/SCHEMA-POLICY.md) once Item 15 lands.
3. Test the gate by simulating a tag locally: `git tag v1.0.0-test` and inspecting the workflow output. Do **not** push the test tag. (The maintainer can also dry-run by editing the workflow on a branch and pushing to verify the step parses cleanly; revert before merging.)

**Acceptance (verbatim from Item 16):**

- A tag created without a matching CHANGELOG entry fails CI.
- A tag created with a matching CHANGELOG entry passes CI.

**Tests:**

- **Manual verification** before the v1.0 stamp: ensure `## [1.0.0]` exists in CHANGELOG before pushing the tag. The gate will fail-fast otherwise and the maintainer can amend before re-pushing.
- **Optional dry-run** on a feature branch with a throwaway tag to confirm the grep regex matches the actual CHANGELOG format. The current `CHANGELOG.md` uses `## [X.Y.Z]` (Keep-a-Changelog convention) — confirm the regex still matches if the format shifts.

### 3.2 Item 14 — CSS withdrawal-vignette scoping verification

**Context (verbatim from Item 14):**

> The withdrawal vignette is mounted to Foundry's global `#interface` selector. If `styles/withdrawal-vignette.css` writes rules against `#interface` directly (rather than against a module-scoped descendant element appended to `#interface`), it risks collision with other modules and themes that touch the same selector. This is a small verification task with a potential small fix.

**Scope:** Read [styles/withdrawal-vignette.css](../../../styles/withdrawal-vignette.css) and confirm every selector targets a module-scoped descendant (`.fishut-vignette`), not `#interface` itself. Run a live-world test with at least one UI theme module installed to confirm no visual collision. Refactor only if the test fails.

**Out of scope:** Refactoring the vignette mechanism itself (no opacity/color/transition changes). Refactoring `styles/module.css` selectors — separate concern unless the live-world test surfaces a collision there.

**Implementation:**

1. **Verification pass.** Read every selector in [styles/withdrawal-vignette.css](../../../styles/withdrawal-vignette.css). Confirm all selectors target `#interface > .fishut-vignette` (current state per §2.1) or a similar scoped descendant. No selector should target `#interface`, `body`, `#ui-left`, `#ui-right`, `#chat`, or any other Foundry global element directly. Same pass over [styles/module.css](../../../styles/module.css) — flag any global-selector usage.
2. **Live-world cross-module test.** Spin up a clean v13 world with:
   - dnd5e 5.2.5
   - substances-and-paraphernalia (current branch)
   - DAE, Midi-QoL, Token Magic FX (hard-required)
   - At least one UI-theming module — recommended: [Ernie's Modern Layout](https://foundryvtt.com/packages/ernies-modern-layout) and [Minimal UI](https://foundryvtt.com/packages/minimal-ui) — install whichever is currently maintained for v13.
   Apply a withdrawal AE to a controlled actor. Verify:
   - The red vignette renders on the screen edge as expected.
   - No theme element is occluded, recolored, or has its layout disrupted by the vignette element.
   - Disabling the theme module removes its styles cleanly; the vignette continues to render correctly.
   - Disabling the vignette (removing the withdrawal AE) leaves the theme intact.
3. **Refactor (only if step 2 fails).** If a collision is found, increase selector specificity:
   - First try `#interface > .fishut-vignette[data-active="true"]` (already partially in place).
   - If insufficient, mount the vignette under `<body>` directly with an explicit ID (`#fishut-vignette`) and rewrite the CSS accordingly. Update [scripts/ui/withdrawal-vignette.js](../../../scripts/ui/withdrawal-vignette.js) `refreshVignette` to mount to `document.body` instead of `document.getElementById("interface")`. Keep the `data-active` toggle behavior.
4. Document the live-world test result in the v1.0 CHANGELOG entry under "Verified" (or add a one-line `tested with: <module list>` note to the release-notes).

**Acceptance (verbatim from Item 14):**

- No selector in [styles/withdrawal-vignette.css](../../../styles/withdrawal-vignette.css) writes rules to a global Foundry element.
- Live-world test with a UI theme installed shows no collision.

**Tests:**

- **Manual** live-world cross-module test as described in implementation step 2. No automated regression — Quench cannot script a visual collision test.

### 3.3 Item 15 — Schema-freeze policy

**Context (verbatim from Item 15):**

> Every minor version 0.2 through 0.6 has shipped breaking schema changes, which is normal for pre-1.0 modules but unsustainable post-1.0. The SPEC already declares the intended migration policy (sheet-level read-with-default, no document-rewriting). Codify the schema-freeze decision in a written policy document before v1.0 releases.

**Scope:** A written policy doc at [docs/SCHEMA-POLICY.md](../../../docs/SCHEMA-POLICY.md). A link from [CONTRIBUTING.md](../../../CONTRIBUTING.md). A reference in the v1.0 release notes.

**Out of scope:** Retroactively applying the policy to pre-v1.0 versions. Implementing any enforcement mechanism — the policy is a written commitment, enforced by maintainer discipline.

**Implementation:**

1. Create [docs/SCHEMA-POLICY.md](../../../docs/SCHEMA-POLICY.md) covering:
   - **Scope.** Applies to `flags["substances-and-paraphernalia"].*` fields read or written by any code in `scripts/`, and to enum values in [scripts/data/schema.json](../../../scripts/data/schema.json).
   - **Rules from v1.0 onward.**
     - Adding a new optional field: **allowed** in any minor version. Readers default missing values via sheet-level read-with-default in [scripts/data/flag-schema.js](../../../scripts/data/flag-schema.js).
     - Adding a new enum value: **allowed** in any minor version. Authors writing the new value confirm via in-world testing that older worlds tolerate the unknown value (Foundry stores it; pre-update code paths fall through to the default branch).
     - Renaming a field or enum value: **forbidden** without a major-version bump.
     - Removing a field or enum value: **forbidden** without a major-version bump.
     - Changing the semantic meaning of an existing field: **forbidden** without a major-version bump.
   - **Migration mechanism.** Sheet-level read-with-default is the path. The `MIGRATORS` skeleton in [scripts/migrations.js](../../../scripts/migrations.js) stays empty unless an explicit one-time semantic-rewrite is needed (in which case the migrator must be idempotent and well-tested).
   - **`schemaVersion` semantics.** Bump on every minor version that adds fields. Readers ignore the version number except for logging; the field's purpose is forensic, not gating.
   - **Out-of-band exceptions.** Bug fixes that touch the schema's behavior but not its shape (e.g., a default-value correction) are allowed in patch versions with a clear CHANGELOG note.
2. Add a link to the policy from [CONTRIBUTING.md](../../../CONTRIBUTING.md). If `CONTRIBUTING.md` does not exist yet, create it with at minimum:
   - A "Release process" sub-section referencing Item 16's CHANGELOG gate.
   - A "Schema changes" sub-section linking to `docs/SCHEMA-POLICY.md`.
   - A pointer to CLAUDE.md and the per-release specs in `docs/superpowers/specs/`.
3. Reference the policy in the v1.0 release notes (`## [1.0.0]` CHANGELOG entry).
4. Cross-reference from [CLAUDE.md](../../../CLAUDE.md) under the existing "Memory + roadmap context" bullet on schema migration: replace the current "Schema migration framework is explicitly out of scope" line with a pointer to `docs/SCHEMA-POLICY.md` (the policy now formalizes what was previously an implicit rule).

**Acceptance (verbatim from Item 15):**

- [docs/SCHEMA-POLICY.md](../../../docs/SCHEMA-POLICY.md) exists.
- [CONTRIBUTING.md](../../../CONTRIBUTING.md) links to it.
- v1.0 release notes reference it.

**Tests:** No code-level test. Verify by reading.

### 3.4 Item 17 — Content warning + safety-tools recommendation + Foundry submission

**Context (verbatim from Item 17):**

> The module ships substances modeled on real-world illicit drugs, with names suggesting amphetamines, caffeine pills, and psychedelics. This is fine creatively and within the bounds of published 5e content, but the README and GM Guide should have a clear up-front "for groups where this content is welcome" note with a recommendation to use safety tools (Lines and Veils, X-Card, or equivalent) at the table. The Foundry package registry may also require age flagging at submission time depending on their content guidelines.

**Scope (PR portion):** Content-warning text + safety-tools recommendation in the README and the GM Guide journal pointer page.

**Scope (manual portion):** At v1.0 release time, read Foundry's package-registry content guidelines and apply whatever `module.json` age-flag or content-tag is required at that time.

**Out of scope:** Removing or softening any existing content. The module's content is intentionally adult-oriented.

**Implementation (PR portion):**

1. Add a content-warning section to [README.md](../../../README.md) immediately above the "What ships" section (or whatever the first content-overview section is). Wording (lifted from Item 17 action 1, lightly polished):
   > ## Content warning
   >
   > This module models illicit substances, addiction, withdrawal, and overdose. The content is intended for adult tables and for groups where this material is welcome.
   >
   > Before introducing substances into your campaign, use a safety tool to confirm consent at the table. Recommended:
   > - [Lines and Veils](https://rpg.stackexchange.com/q/2802) — pre-game agreement on hard nos (lines) and fade-to-black topics (veils).
   > - [X-Card](http://tinyurl.com/x-card-rpg) — in-the-moment opt-out card.
   > - [Script Change](https://briebeau.com/thoughtypod/games/script-change/) — pause / rewind / fast-forward signals.
   >
   > These tools are not a substitute for session-zero conversations. Use them in addition to direct check-ins with your players.
2. Adapt the same wording to a Foundry-journal-page HTML block. Add the page to the GM Guide journal source: [_source/fishut-journals/Substances_and_Paraphernalia___GM_Guide_fhJrnlGmGuide001.json](../../../_source/fishut-journals/Substances_and_Paraphernalia___GM_Guide_fhJrnlGmGuide001.json). Place it as the first page so a GM opening the journal sees the warning before any mechanical content. Rebuild via `npm run pack`.

**Implementation (manual submission portion — release-time, post-PR):**

4. At v1.0 release time, after Items 14, 15, 16, and the PR portion of 17 are merged and the `v1.0.0` tag is pushed (and the release workflow has produced `module.json` + `module.zip` assets):

   - If a flag is required, add it to [module.json](../../../module.json) and push a patch tag (`v1.0.1`) with the addition; the gate from Item 16 still requires a CHANGELOG entry for the patch.
   - Submit the module through Foundry's package-submission form using the `module.json` URL pattern documented in the existing module.json (`releases/latest/download/module.json`).

**Acceptance (verbatim from Item 17):**

- README has a content-warning section above the "What ships" section.
- GM Guide journal pointer has the equivalent.
- v1.0 submission complies with Foundry's content guidelines. *(Verified at submission time.)*

**Tests:**

   - CHANGELOG CI gate (Added — release hygiene).
   - CSS scoping verification (Verified — no behavior change unless live-world test surfaced a collision; in that case Changed).
   - Schema-freeze policy adopted (Added — `docs/SCHEMA-POLICY.md`).
   - Content warning + safety-tools recommendation (Added — documentation).
   - Reference to the schema policy as the post-v1.0 commitment.
6. **Manual Foundry submission (Item 17 manual portion).** After the release artifacts are produced. Follow §3.4 implementation step 4.

Each PR ships with: lint green, validate green, unit tests green, Quench suite green (no v1.0 work adds new tests; existing suite must stay green).

## 6. Out of scope (explicit)

These items are tempting but **do not land in v1.0**, regardless of how small they seem:

- **Mechanical changes.** v0.7 thesis.
- **Authoring-UX changes.** v0.8 thesis.
- **Compendium content changes.** v0.9 thesis.
- **Document-level schema migrator.** Per CLAUDE.md, [docs/SCHEMA-POLICY.md](../../../docs/SCHEMA-POLICY.md) (Item 15), and roadmap §7 — sheet-level read-with-default is the migration path. `MIGRATORS` stays empty.
- **Enforcing the schema policy in code.** The policy is a written commitment.
- **Auto-generating CHANGELOG entries.** Item 16 is a gate, not a generator.
- **Pre-committing to a specific `module.json` content-flag.** Foundry's guidelines may evolve; the choice is deferred to submission time per §2.4.
- **Removing the substring-name fallback** (per v0.7 §2.2).
- **Reintroducing the deleted 3-dot-menu authoring form.**
- **GM-only paths for any player-facing UI.**
- **World settings whose off-state nobody wants.** Per CLAUDE.md.
- **Downgrading compendium pack ownership.** Per CLAUDE.md and roadmap §6.
- **Adding tests for unmodified code paths** to inflate v1.0's diff. The release is hygiene; the diff stays small.

## 7. Release-thesis check

Every item lands inside the "stabilize and submit" thesis:

- Item 16 — release hygiene gate; catches CHANGELOG omissions before they ship.
- Item 14 — release hygiene verification; confirms the vignette plays nice with the wider Foundry ecosystem before submission.
- Item 15 — stabilize the schema commitment; signals to GMs and contributors what v1.0 promises about backward compatibility.
- Item 17 — submission prerequisite; the content warning and safety-tools recommendation are table-stakes for an adult-themed module entering the registry.

Nothing in this release is mechanical correctness (v0.7), authoring clarity (v0.8), or content rebuild (v0.9). v1.0 is exclusively the polish-and-publish release.
