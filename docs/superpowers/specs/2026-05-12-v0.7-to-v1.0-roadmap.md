## 4. Sequencing rationale

The non-obvious decisions encoded here:

### 4.1 Items 3 + 5 + 6 ship together in v0.7

The original review proposed Item 6 (Overdose-Tolerance interaction) in v0.8. Pulling it forward into v0.7 with Item 5 (Tolerance bounds, its soft prerequisite) avoids re-tuning churn: Item 3's "voluntary abstain failure triggers consumption" path runs through the full post-use addiction + overdose chain, so the playtest signal for Item 3 is only meaningful once Item 6 is live. Bundling the three avoids landing Item 3, tuning it against unmodulated overdose math, then re-tuning a release later when Item 6 changes the math underneath.

The implementation seam in `scripts/hooks/addiction.js` and `scripts/hooks/overdose.js` is two independent `dnd5e.postUseActivity` listeners, so Items 3 and 6 don't merge-collide structurally — they touch different files. The reason to bundle them is the playtest cycle, not the code path.

### 4.2 Item 13 (DC scaling) moves from v0.9 to v0.8

The original review placed Item 13 with the v0.9 documentation lift. Item 13 is small (a wiki sub-section + a hint string under Save DC) and shares v0.8's "authoring clarity" thesis. Moving it to v0.8 rebalances the release weights without compromising either thesis. It also unblocks Item 12 — the compendium rebuild needs the DC-tier convention written down before it consumes it.

### 4.3 Item 1 is split across v0.7 and v0.9

The schema continues to allow any save ability per substance. The UI hint string + wiki sub-section + authoring convention are documentation work that lands in v0.7 (cheap to ship, no content dependencies). The shipped-content update (every mind-altering substance switches to Wisdom) waits for the v0.9 compendium rebuild because every shipped substance is being rewritten anyway and pre-touching content in v0.7 is waste.

### 4.4 Item 11 is split across v0.8 and v0.9

The 2024 language audit covers four surfaces: `lang/en.json` (UI strings), `templates/*.hbs` (chat card templates), `_source/**/*.json` (shipped item/AE descriptions), and `_source/fishut-journals/*.json` (in-world journal). The first two are stable and audit-once. The third is rewritten in Item 12 anyway, so auditing it pre-rebuild is waste. v0.8 ships the audit pass over the stable surfaces plus a warn-only invariant in `tools/validate-content.mjs`. v0.9 ships the rest as a natural byproduct of Item 12 and flips the invariant to error-blocking once content is clean.

### 4.5 Item 12 has a roster shell in v0.9, but a sub-spec deferred to v0.9 kickoff

The v0.9 spec defines Item 12's matrix shape (3×3 setting × category, ≥2 substances + ≥1 paraphernalia per cell, invented examples for the manuals not reused from the matrix). Per-substance authoring plans are deferred to a sub-spec written at v0.9 kickoff. Rationale: v0.7/v0.8 playtest may surface tuning lessons that reshape the rebuild plan; locking per-substance detail now is premature.

### 4.6 Item 17's Foundry submission is a manual step, not a code PR

The README + journal content-warning text ships as a normal v1.0 PR. The Foundry package registry submission itself happens at v1.0 release time and is a manual step (read the current Foundry submission guidelines, set whatever `module.json` mature-content flag exists then, submit). The Foundry guidelines may evolve between now and v1.0, so pre-committing to a specific `module.json` change now would be guessing.

## 7. Out-of-scope across all four releases

Items the implementing agent might be tempted to introduce but should not, regardless of release:

- **Document-level schema migrators.** SPEC and CLAUDE.md both call this out. Sheet-level read-with-default is the migration path. The `MIGRATORS` skeleton in `scripts/migrations.js` stays empty unless an explicit semantic-rewrite is needed.
- **Reintroducing the deleted 3-dot-menu authoring form.** Details-tab authoring is the only authoring surface.
- **GM-only paths for gating dialogs, override buttons, or save-bypass chat cards.** All player-facing UI stays visible to all users.
- **Removing the AE substring-name fallback after Item 2 lands.** Substring matching stays as a warn-logged fallback for hand-authored AEs.
- **World settings whose off-state nobody wants.** Per CLAUDE.md: prefer baked-in behavior.
- **Downgrading compendium pack ownership.**
