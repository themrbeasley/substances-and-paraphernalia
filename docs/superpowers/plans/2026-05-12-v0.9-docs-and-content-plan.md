```markdown
## Part 2 — Reference

Dip into a sub-section by subsystem. They are not ordered to be read top-to-bottom.

### Altered AE reference

- AE Changes table conventions (mode 0 CUSTOM for `macro.tokenMagic` rows;
  mode 5 OVERRIDE for direct flag writes; mode 2 ADD for numeric bonuses).
- TMFX integration: the `macro.tokenMagic` Change row pattern is DAE-driven —
  not a custom hook. Cross-reference the project's CLAUDE.md section
  "TMFX integration is DAE-driven, not a custom hook" for the apply/remove
  semantics. Preset names follow `fishut-tmfx-{setting}-{category}`.
- Recommended duration formats: round counts via `system.duration`; descriptive
  text in the AE description body.

### Addiction AE reference

- The Addiction AE template lives on the substance item and is pointed to by
  `flags["substances-and-paraphernalia"].addiction.addictionEffectId`.
- Poisoned-coupling interaction: the addiction AE may carry `statuses: ["poisoned"]`
  directly, or you may let the coupling mode handle it. See
  [Poisoned Coupling](Poisoned-Coupling.md) for the three modes.
- Name contract: the AE name must contain `addict` (case-insensitive). The
  module also reads the `aeRole: "addiction"` flag added in v0.7 — set it
  alongside the substring.

### Withdrawal AE reference

- Vignette color: author a Change row with
  `key: "flags.substances-and-paraphernalia.vignetteColor"`,
  `mode: 5` (OVERRIDE), `value: "<#hex>"`, `priority: 20`.
- Default fallback color is `#a02020` (red bloom).
- Do **not** put the `poisoned` status on the withdrawal AE — the addiction AE
  already carries it. Duplicating causes a warn from `npm run validate`.
- Name contract: must contain `withdraw`; the v0.7 `aeRole: "withdrawal"` flag
  is the primary lookup key.

### Tolerance reference

- The three knobs:
  - `attenuateAltered.{durationFactor, modifierFactor}` — what each tolerance
    stack does to the Altered AE.
  - `addictionDcBump` — how much each stack raises the save DC.
  - `withdrawalAmplify.{durationFactor, modifierFactor}` — how each stack
    extends or strengthens withdrawal.
- Bounds: caps default to 6 stacks for fully-progressive substances and
  3 stacks for medium-burn substances. Override via `tolerance.caps`. See
  [Mechanics](Mechanics.md) for tier guidance.
- When to leave tolerance off: teaching examples (this one), one-shot
  legendary doses, or substances that should *not* desensitize.

### Overdose reference

- `chancePercent` semantics: integer 1..100, the percent chance of an overdose
  card firing on each consumption.
- Description-only chat card: the `description` field is the chat-card body;
  no mechanical AE applies unless you author one explicitly via the Overdose
  AE template.
- Tolerance interaction (v0.7): set `overdose.toleranceInteraction` to
  `dampen`, `amplify`, or `none` to control how the d100 chance shifts as
  tolerance stacks accumulate.
```

- [ ] **Step 2: Write the Common Pitfalls sidebar (Part 3)**

Append below the reference:

```markdown
## Part 3 — Common pitfalls

A short sidebar of mistakes the author keeps making. These are not bugs —
they are authoring footguns.

- **Forgetting to set the save ability for mind-altering substances.** The
  per-category default is Wisdom; if you forget the picker, Constitution
  ships and the save reads wrong for the category. The Details-tab hint
  string flags the per-category default.
- **Leaving a withdrawal AE template empty.** A withdrawal AE with no
  Changes still applies — but it does nothing, which silently breaks the
  withdrawal flavor. At minimum carry the vignette-color Change.
- **Setting Tolerance knobs without thinking about cumulative bite.** Three
  stacks at `durationFactor: 0.5` halves the Altered AE twice; the fourth
  stack barely matters. Tune so the *cumulative* curve is interesting, not
  the per-stack number.
- **Accidentally making consumption a pure-loss.** If Addiction DC + Withdrawal
  Mod outweigh the Altered benefit at the player's level, the substance is
  punitive and never used. Sanity-check by playtesting on a level-1 character
  with average ability scores.
```

```markdown
## Part 2 — Reference

### Subtype model

- Subtypes are a kebab-case open enum. Built-ins ship in
  [scripts/data/schema.json](../../scripts/data/schema.json) under
  `paraphernaliaSubtypes`.
- Authors can extend the list per-world via the **Paraphernalia Subtype Manager**
  settings sub-menu. See [Paraphernalia Subtype Manager](Paraphernalia-Subtype-Manager.md).
- A subtype's only job is grouping for the Details-tab dropdown — it has no
  mechanical effect.

### Admin-type matching

- The consumption gate keys off the dnd5e Poison subtype on the *substance*
  (`system.type.subtype` ∈ `contact | ingested | inhaled | injury`).
- Each paraphernalia advertises an `appliesTo` array of admin types via the
  Details-tab checkboxes. The gate passes when the actor owns at least one
  ready paraphernalia whose `appliesTo` covers the substance's admin.
- See CLAUDE.md's "Admin-type gate" section for the underlying pure helper.

### Save bypass authoring

The four tiers, in resolution order (highest wins):

1. **`auto-pass`** — bypasses the save entirely. Use for setpiece legendary
   gear; expect to gate behind `usesPerDay`.
2. **`reroll-on-fail`** — rolls the save once; if it fails the DC, re-rolls
   once with clean configuration (no advantage, no bonus). The reroll's
   result is canonical.
3. **`advantage`** — rolls the save with advantage.
4. **`+N`** — adds a flat numeric bonus to the save. Multiple `+N` bypasses
   sum.

Authoring conventions:

- Live on the paraphernalia as a `transfer: true` Active Effect.
- Flag block on the AE: `modifier: { kind: "bypass", type, ..., appliesTo: [<admin>...] }`
  plus the `aeRole: "bypass"` flag from v0.7 (Item 2).
- Per-day uses ride on dnd5e's native `system.uses.recovery` set to
  `[{ period: "day", type: "recoverAll" }]` on the paraphernalia item.
  `validate-content` requires this when an embedded bypass AE declares
  `usesPerDay`.
- See [Save Bypass Tiers](Save-Bypass-Tiers.md) for the full resolution
  pipeline and worked examples.

### Readiness gating

A paraphernalia is considered *ready* when:

- Its item is owned by the actor.
- Equipment paraphernalia must be **equipped** (and **attuned** if the item
  requires attunement).
- Consumable paraphernalia must have **uses remaining**
  (`system.uses.value > 0` or equivalent).

The gate only counts ready paraphernalia. A pipe in the backpack does not
satisfy the gate; a pipe equipped in a hand slot does.
```

```markdown
## Part 3 — Common pitfalls

- **Forgetting to set `appliesTo`.** A paraphernalia with empty `appliesTo`
  matches no substance. The gate will block every substance even when the
  paraphernalia is equipped.
- **Authoring a bypass AE without `transfer: true`.** Without `transfer`, the
  AE stays on the item and never lands on the actor; the bypass pipeline
  finds nothing.
- **Picking a bypass tier higher than the substance design warrants.** An
  `auto-pass` bypass renders the substance's save irrelevant. Default to
  `+N` or `advantage` and reserve `reroll-on-fail` / `auto-pass` for
  setpiece legendary gear with `usesPerDay`.
```
