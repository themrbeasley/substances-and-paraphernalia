```markdown
## DC Scaling Across Tiers

The module accepts any save DC, but shipped substances follow this tier-aligned convention:

| Character tier | Levels | Recommended DC range | Pass rate (vs +1 Con) | Pass rate (vs +5 Con) |
|---|---|---|---|---|
| 1 | 1–4 | 12–14 | ~50% | ~70% |
| 2 | 5–10 | 15–17 | ~35% | ~55% |
| 3 | 11–16 | 18–20 | ~20% | ~40% |
| 4 | 17–20 | 21+ | ~10% | ~25% |

The intent is **tier-1-to-tier-2 by default** — most shipped substances sit at DC 12–16 so they remain a real but manageable narrative threat through most campaigns. DCs of 17+ are reserved for tier-3-plus setpiece substances (alchemical horrors, story-critical addictions, etc.) where the addiction itself is a deliberate antagonist.

This convention matches the 2024 SRD poison rules and gives the v0.9 compendium rebuild a known target. Authors are free to override per-substance — the schema accepts any DC. The Save DC field in the Details tab shows a hint with the same ranges to keep the convention discoverable at the authoring surface.
```

# PR 2 — Item 4: Withdrawal duration preview + subsystem visibility

## Task 2.1: Add `previewWithdrawalDuration` to `scripts/data/withdrawal.js`

**Files:**
- Modify: `scripts/data/withdrawal.js`

- [ ] **Step 1: Add the new export**

Append to `scripts/data/withdrawal.js` after `computeRestsRemaining`:

```js
/**
 * Cosmetic-only preview of withdrawal duration for the Details-tab authoring
 * surface. Reuses computeRestsRemaining so the UI preview can never drift from
 * the engine's actual computation.
 *
 * @param {number} withdrawalMod   The substance's withdrawalMod (Details-tab input).
 * @param {number} [assumedConMod] Defaults to 0 — the author has no character context
 *                                 at item-edit time, so the preview must disclose
 *                                 the assumption in its display string.
 * @returns {number}               Long rests of withdrawal owed (>= 1).
 */
export function previewWithdrawalDuration(withdrawalMod, assumedConMod = 0) {
  return computeRestsRemaining(withdrawalMod, assumedConMod);
}
```

The function is a one-line passthrough — its only purpose is to declare the assumed-Con-+0 contract at the call site so the UI doesn't accidentally call `computeRestsRemaining` with a real actor's Con and mislead the author.

The `withdrawal` subsystem is currently its own peer fieldset (not nested inside addiction). Both collapses must pass for the inner fields to be visible. The simpler reading of the spec is "treat the existing four fieldsets as four collapse blocks, each gated by its own enabled flag." ## Tuning Withdrawal Duration

```markdown
The number of long rests an addiction's withdrawal AE persists is computed at the moment of addiction onset, not at authoring time. The formula is:

```

restsRemaining = max(withdrawalMod − ConMod, ⌈withdrawalMod / 2⌉)

```

Clamped to a minimum of 1. The `⌈withdrawalMod / 2⌉` term is the **floor clamp** — it prevents a high-Con character from waving off withdrawal entirely.

The Details-tab authoring surface shows a live preview that **assumes Con +0**. The disclosure text says "varies per character" because a tough actor will shave 1–3 rests off the preview and a sickly actor will add rests. To see the real range for a specific party, run the worked examples mentally against your players' Con modifiers.

### Worked examples

| withdrawalMod | Con +0 | Con +3 | Con +5 |
|---|---|---|---|
| 2 | 2 | 1 | 1 |
| 4 | 4 | 2 | 2 |
| 6 | 6 | 3 | 3 |

Note that Con +5 against withdrawalMod 6 still owes 3 rests — that's the floor clamp. A tier-1 substance with `withdrawalMod: 2` is a single long rest for any Con +3-or-better PC; a tier-3 setpiece substance with `withdrawalMod: 6` always costs 3 rests minimum.

### Choosing withdrawalMod

- **1–2:** Trivial / narrative; one rest of "feeling rough."
- **3–4:** Moderate; a tier-1-or-2 substance whose withdrawal is a real obstacle.
- **5–6:** Heavy; tier-3-plus substance whose withdrawal is itself a plot driver.
- **7+:** Use at your own risk; floor clamp keeps it long even for tough PCs, and PCs without Con investment will drag the cycle out across multiple sessions.
```

```js
    id: "once-per-day",

    message: 'use "regains all expended uses at dawn" or "you can\'t use this again until you finish a Long Rest"',

    id: "become-condition",

    message: 'capitalize "Long Rest" / "Short Rest"',
    appliesTo: "text-content-only",
  },
];

/**
 * Scan `text` and return warn-level findings.
 *
 * @param {string} text
 * @param {{sourcePath?: string, mode?: "text-content-only" | "any"}} [opts]
 * @returns {Array<{ruleId: string, match: string, sourcePath?: string, message: string}>}
 */
export function checkLanguagePhrasing(text, opts = {}) {
  const { sourcePath = "<input>", mode = "any" } = opts;
  if (typeof text !== "string" || text.length === 0) return [];
  const findings = [];
  for (const rule of RULES) {
    if (rule.appliesTo === "text-content-only" && mode !== "text-content-only") continue;
    const m = text.match(rule.pattern);
    if (!m) continue;
    findings.push({
      ruleId: rule.id,
      match: m[0],
      sourcePath,
      message: rule.message,
    });
  }
  return findings;
}

export const _LANGUAGE_RULES = RULES;
```

The `appliesTo: "text-content-only"` flag is the spec-mandated filter from §3.3 action 4 ("filter the invariant to text-content paths"). The CLI wrapper in Task 3.4 sets `mode: "text-content-only"` only when scanning `lang/en.json` string **values** and `templates/*.hbs` text content — never when scanning `system.type.subtype: "poison"` (which is a data field, not prose) or any other field whose value is enum-typed.

```js
// 2024 language audit (warn-only in v0.8; flips to error-blocking in v0.9).
async function scanLangStringsForPhrasing() {
  const path = resolve(ROOT, "lang/en.json");
  let json;
  try {
    const raw = await readFile(path, "utf8");
    json = JSON.parse(raw);
  } catch (e) {
    warnings.push(`lang/en.json: failed to read for language scan: ${e.message}`);
    return;
  }
  for (const [key, value] of Object.entries(json)) {
    if (typeof value !== "string") continue;
    const findings = checkLanguagePhrasing(value, {
      mode: "text-content-only",
      sourcePath: `lang/en.json:${key}`,
    });
    for (const f of findings) {
      warnings.push(`${f.sourcePath} [${f.ruleId}]: "${f.match}" — ${f.message}`);
    }
  }
}

async function scanTemplatesForPhrasing() {
  const templatesRoot = resolve(ROOT, "templates");
  let entries;
  try {
    entries = await readdir(templatesRoot, { withFileTypes: true, recursive: true });
  } catch (e) {
    warnings.push(`templates/: failed to enumerate: ${e.message}`);
    return;
  }
  for (const entry of entries) {
    if (!entry.isFile() || !entry.name.endsWith(".hbs")) continue;
    const path = resolve(entry.parentPath ?? entry.path, entry.name);
    const relPath = path.slice(ROOT.length + 1).replace(/\\/g, "/");
    let raw;
    try {
      raw = await readFile(path, "utf8");
    } catch (e) {
      warnings.push(`${relPath}: failed to read for language scan: ${e.message}`);
      continue;
    }
    // Strip Handlebars expressions {{...}} so we don't flag the JSON-y inner
    // syntax. Replace with a single space so word boundaries on the surrounding
    // text are preserved.
    const stripped = raw.replace(/{{[\s\S]*?}}/g, " ");
    // Strip HTML attributes inside tags (we only want visible text). A coarse
    // approximation: keep only text between > and <. False negatives are OK
    // since this is warn-only; false positives on attribute values would noise
    // the audit beyond usefulness.
    const visible = stripped.replace(/<[^>]+>/g, " ");
    const findings = checkLanguagePhrasing(visible, {
      mode: "text-content-only",
      sourcePath: relPath,
    });
    for (const f of findings) {
      warnings.push(`${f.sourcePath} [${f.ruleId}]: "${f.match}" — ${f.message}`);
    }
  }
}

await scanLangStringsForPhrasing();
await scanTemplatesForPhrasing();
```

The `readdir(..., { recursive: true })` API needs Node 20+ — `package.json` already declares `"engines": { "node": ">=20" }`, so this is safe.

The new scans append to `warnings`, never to `errors`. The existing exit logic in `tools/validate-content.mjs` only exits non-zero when `errors.length > 0`, so the warn-only contract holds without further change.
