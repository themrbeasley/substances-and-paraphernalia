- [ ] **Step 2: Insert the CHANGELOG-verification step**

Edit [.github/workflows/release.yml](../../../.github/workflows/release.yml). After the `Unit tests` step (currently:
```yaml
      - name: Unit tests
        run: npm run test:unit
```
) and before the `Pack compendia` step, insert:

```yaml
      - name: Verify CHANGELOG entry exists for tag
        run: |
          VERSION="${GITHUB_REF_NAME#v}"
          if ! grep -qE "^## \[${VERSION}\]" CHANGELOG.md; then
            echo "::error::CHANGELOG entry missing for tag ${GITHUB_REF_NAME}. Add a '## [${VERSION}]' section to CHANGELOG.md before tagging."
            exit 1
          fi
          echo "CHANGELOG entry found for ${GITHUB_REF_NAME}."
```

Rationale: positioning before `Pack compendia` means a missing CHANGELOG fails the workflow before any artifact is produced â€” no zip, no release object, no asset upload. The `${GITHUB_REF_NAME#v}` shell parameter expansion strips a single leading `v` from `v1.0.0` â†’ `1.0.0`, matching the Keep-a-Changelog `## [1.0.0]` heading format (CHANGELOG.md line 10: `## [0.6.0] â€” 2026-05-11`). The `grep -qE` with `^## \[${VERSION}\]` is anchor-strict â€” only matches at line start and requires the literal `[` immediately after `## `, so it won't falsely match `### [0.6.0]` or in-paragraph text.

- [ ] **Step 1: Create the policy document**

Create [docs/SCHEMA-POLICY.md](../../../docs/SCHEMA-POLICY.md) with this content:

```markdown
# Schema-freeze policy

**Status:** Adopted at v1.0.

This document codifies the backward-compatibility commitment that
`substances-and-paraphernalia` makes to GMs and contributors from v1.0
onward. Pre-v1.0 versions are not covered â€” every minor bump from 0.2 to
0.6 shipped breaking schema changes, which is normal for pre-1.0 modules.
That phase is now over.

## Scope

This policy governs:

1. **Item-flag fields** under `flags["substances-and-paraphernalia"].*` â€”
   every field read or written by code in `scripts/`.
2. **Actor-flag fields** under `flags["substances-and-paraphernalia"].*`
   on actors (`withdrawal`, `vignetteColor`, the addiction-state block).
3. **Active Effect flags** under `flags["substances-and-paraphernalia"].*`
   on AEs (`modifier`, `aeRole`, `sourceSubstanceId`, `overdose`, `tolerance`).
4. **Enum values** in [scripts/data/schema.json](../scripts/data/schema.json)
   (`kind`, `category`, `setting`, `administration`, `paraphernalia.subtypes`
   built-ins, `modifier.types`, `addiction.save.ability` defaults, etc.).
5. **`schemaVersion`** under `flags["substances-and-paraphernalia"].schemaVersion`
   on items.

It does **not** govern:

- Foundry's own document shape (`system.*`, `effects.*` top-level fields).
  Those follow dnd5e's and Foundry's own compatibility policies.
- TMFX preset names. Those are registered into TMFX's `tmfx-main` library
  and follow TMFX's compatibility, not ours.
- World-setting keys under the `substances-and-paraphernalia` namespace.
  Settings keys are intentionally stable but renames are allowed with a
  one-version deprecation note in `CHANGELOG.md`.

## Rules from v1.0 onward

### Allowed in any minor version

- **Adding a new optional field.** Readers default missing values via
  sheet-level read-with-default in
  [scripts/data/flag-schema.js](../scripts/data/flag-schema.js). Authors
  treat the field as optional and writers may omit it.
- **Adding a new enum value.** Authors writing the new value confirm via
  in-world testing that older worlds tolerate the unknown value (Foundry
  stores it; pre-update code paths fall through to a default branch â€” see
  [scripts/data/admin-match.js](../scripts/data/admin-match.js) and
  [scripts/data/modifier-pipeline.js](../scripts/data/modifier-pipeline.js)
  for the existing patterns).
- **Bug fixes that don't change shape.** A default-value correction, a
  parsing bug, a typo in a `labelKey` â€” all allowed in patch versions with
  a clear `CHANGELOG.md` note.

### Forbidden without a major-version bump

- **Renaming a field.** No silent rewrites.
- **Renaming an enum value.** No silent rewrites.
- **Removing a field.** No silent rewrites.
- **Removing an enum value.** No silent rewrites.
- **Changing the semantic meaning of an existing field.** Same key, different
  contract â€” counts as a rename for the purposes of this policy.

A major-version bump (v1.0 â†’ v2.0) is a deliberate compatibility break.
It is announced in advance, documented in `CHANGELOG.md`, and the migration
path is documented in a release note.

## Migration mechanism

Sheet-level read-with-default in
[scripts/data/flag-schema.js](../scripts/data/flag-schema.js) is the
canonical migration mechanism. Readers tolerate missing fields by
substituting documented defaults. The `MIGRATORS` skeleton in
[scripts/migrations.js](../scripts/migrations.js) **stays empty** unless an
explicit one-time semantic-rewrite is needed â€” in which case the migrator
must be idempotent and well-tested.

Document-level migrators were intentionally not adopted pre-v1.0 (see
[ROADMAP.md](../ROADMAP.md) and [CLAUDE.md](../CLAUDE.md)). v1.0 formalizes
that decision: read-with-default scales further than the alternative and
does not require running expensive migrations on every world load.

## `schemaVersion` semantics

`flags["substances-and-paraphernalia"].schemaVersion` on items is forensic,
not gating. Readers do not branch on it. Its purpose is to support
post-hoc debugging: a GM reporting an issue can include the
`schemaVersion` value, and the maintainer can correlate the world's state
with the version that originally wrote those flags.

The value bumps on every minor version that adds fields. From v1.0 onward,
the bump is monotonic and additive-only.

## Out-of-band exceptions

- **Pre-v1.0 versions.** This policy does not apply retroactively. Versions
  v0.2 through v0.9 each shipped at least one schema change; that is the
  pre-1.0 contract.
- **Bug fixes that touch behavior but not shape.** Allowed in patch
  versions per "Allowed in any minor version" above.
- **Documentation-only changes to the policy itself.** Editing this file
  to clarify wording is not a policy change.
```
